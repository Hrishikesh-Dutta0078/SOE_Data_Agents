Data Handling & Performance

Server-side aggregation, caching, and optimized data flow

Server-Side Aggregation Pipeline

Currently the client aggregates all data in-browser. With the profiler, aggregates are pre-computed server-side and sent alongside the spec.

BEFORE: Client Does Everything
Server → sends 5,000 raw rows → Client
├ KPI: sum(Amount) in JS
├ Chart: groupBy + aggregate in JS
├ Sparkline: filter + sort in JS
└ Slicer filter: .filter() on 5K rows
⚠ ~200-800ms render lag on large sets
AFTER: Server Pre-Computes
Server → sends tileData[] → Client
├ KPI: { value: 12450000, formatted: "$12.5M" }
├ Chart: 15 pre-grouped rows
├ Sparkline: 12 monthly points
└ Slicer values: pre-fetched
✓ Instant render, minimal payload
Profile & Aggregate Cache
Layer 1: LangGraph State
  • dataProfiles[] stored in workflow state
  • Survives across nodes within a run
  • Refinements read from state directly
  • No re-profiling, no re-sending rows
Layer 2: Session Cache
  • In-memory Map keyed by SQL hash
  • Stores: profile + pre-computed aggregates
  • TTL: session lifetime (cleared on disconnect)
  • Slicer values served from cache instantly
Layer 3: Tile Data API
  • New: /dashboard-data?mode=tile
  • Returns pre-aggregated data per tile
  • Reads from session cache first
  • Falls back to SQL re-execution if evicted
Optimized Refinement Flow
BEFORE
Client sends:
previousDashboardSpec (full JSON)
dashboardDataSources[] (8 rows each)
refinementText
~15-30KB per refinement request
AFTER
Client sends:
previousDashboardSpec (full JSON)
profileCacheKey (hash reference)
refinementText
~3-5KB per refinement request
Server looks up cached profiles by key → feeds them to LLM → no redundant data transfer, no re-profiling
Slicer Loading UX Fix
BEFORE: Fetch on Mount
1. Dashboard opens
2. Empty slicer dropdowns shown
3. N separate DISTINCT queries fire
4. Values trickle in (500-2000ms)
5. Dropdowns populate with flash
AFTER: Pre-Computed in Profile
1. Profiler computes distinct values
2. Values included in dashboard response
3. Dashboard opens with slicers ready
4. Zero additional queries needed
5. Instant, no flash
